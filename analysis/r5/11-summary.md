## Ключевые находки (объясняют прод-симптомы)

### 1. BUG-01 (ВЫСОКИЙ): guided-grab сломан для всех — push_release выбрасывает ответ *arr
`POST /release/push` у Radarr/Sonarr/Lidarr возвращает **массив** (проверено по upstream-исходникам), а клиенты делают `isinstance(result, dict) else {}` → `approved: true` теряется → **каждый** push трактуется как «no explicit approval» (ровно то, что в прод-логе!). Релизы молча уходят в qBittorrent-байпас мимо *arr (без импорт-трекинга/переименования), а юзер жмёт force_grab там, где всё должно было работать само.

### 2. PERF-01 (КРИТИЧЕСКИЙ для Pi): detection-burst — источник одновременных 503
Каждое свободное текстовое сообщение = 3 параллельных тяжёлых `/lookup` (Radarr+Sonarr+Lidarr, каждый ходит во внешний metadata-proxy) + следом Prowlarr-fanout — без семафора, кэша и cooldown. Это единственный код-путь, нагружающий Radarr и Lidarr в один момент — точное совпадение с наблюдаемыми одновременными 503.

### 3. SEC-01 (СРЕДНИЙ): SSRF-фикс 7b3de49 доверяет хосту без порта
Все сервисы живут на одном LAN-IP → доверенным стал ЛЮБОЙ порт этого IP (`:22`, `:6379`…). Нужно сравнение (host, port).

### 4. OBS-01..04: канал диагностики повреждён
stdlib-логгеры (aiogram/httpx) пишут голые строки мимо JSON и мимо маскировщика токена; «Sent completion notification» логируется даже при провале отправки; 89 из 92 `log.error` без traceback.
