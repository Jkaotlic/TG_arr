### SEC-01: HTML injection / message-send failure: torrent.name interpolated unescaped into HTML message in /pause and /resume
- **Файл**: `bot/handlers/downloads.py:132`
- **Проблема**: The bot is constructed with DefaultBotProperties(parse_mode=ParseMode.HTML) (bot/main.py:233), so every Message.answer() without an explicit parse_mode is rendered as HTML. In cmd_pause (downloads.py:132) and cmd_resume (downloads.py:160) the code does await message.answer(f"⏸️ Приостановлен: {torrent.name}") / f"▶️ Возобновлён: {torrent.name}" with torrent.name taken verbatim from the qBittorrent torrent (which itself comes from the release name an indexer/tracker supplied). Release/torrent names very commonly contain '&' (e.g. 'Tom & Jerry', 'S01 [RARBG] <group>'). Concretely: user runs `/pause <8-char-hash-of-a-torrent-named 'A & B'>` -> Telegram receives '⏸️ Приостановлен: A & B', rejects it with 400 Bad Request: can't parse entities, the answer() raises, and the user gets no confirmation (or the handler errors out). A name like 'x</b>' or '<i>' likewise breaks rendering. This is a reliable malfunction on a normal path, and lets remote (indexer-supplied) data inject markup into a privileged bot message.
- **Риск**: Remote indexer/tracker controls torrent names; any '<','>','&' in a name breaks the privileged confirmation message.
- **Решение**: Escape the remote-controlled name before interpolating into an HTML message: import html and use html.escape(torrent.name) at downloads.py:132 and :160 (mirror the existing escaping in music.py / search.py / Formatters._e). Alternatively pass parse_mode=None for these plain-text confirmations.

I grepped downloads.py: every other formatted send uses parse_mode="HTML" explicitly (lines 73,101,199,...), and there is NO 'import html'/html.escape anywhere in the file, so torrent.name is interpolated raw. ### SEC-03: Private-tracker passkey leaked to logs: Radarr/Sonarr push-release response (echoing downloadUrl with passkey) logged in full at INFO
- **Файл**: `bot/services/add_service.py:357`
- **Проблема**: add_service.py logs log.info("Push release result", result=result) at lines 357, 509 and 710. `result` is the raw dict returned by Radarr/Sonarr's /api/v3(or v1)/release/push endpoint (radarr.py:175, sonarr/lidarr equivalents), which echoes the pushed release object including its `downloadUrl`. For private trackers that downloadUrl embeds a secret passkey/apikey (e.g. https://tracker/rss/.../?passkey=SECRET). The only redaction in the logging pipeline is _mask_tokens (bot/main.py:32) which (a) only iterates top-level event_dict values and does NOT recurse into nested dicts/lists, and (b) only substitutes the Telegram bot-token regex — it does not touch passkey/apikey/token query params. The dedicated _mask_url() helper (add_service.py:38) IS used for the download_url= fields but is NOT applied to result=, so the full URL with the passkey is serialized verbatim by JSONRenderer at INFO and persisted to docker json-file logs. Anyone with read access to the container logs (or log aggregation) obtains the tracker passkey, which is a reusable account credential.
- **Риск**: Tracker passkeys are reusable credentials; once in logs they can be exfiltrated to ratio-cheat or get the user's tracker account banned.
- **Решение**: Do not log the raw push response. Either drop result= entirely, log only the safe fields (result.get('approved'), result.get('rejections')), or run the response through a recursive sanitizer that masks downloadUrl/magnetUrl and the _SENSITIVE_QUERY_PARAMS. Additionally harden _mask_tokens to recurse into nested dict/list values and apply URL-query masking, so any future nested URL is redacted by default.
- **Верификация**: CONFIRMED — Reproduced the full leak path in current code. (1) bot/services/add_service.py lines 357, 509, 710 each call log.info("Push release result", result=result) — confirmed verbatim at all three cited lines (Radarr/Sonarr/Lidarr branches).
