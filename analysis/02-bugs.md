### RACE-01: No double-grab guard: rapid double-tap on Confirm/Grab-Best/Force-Grab adds/grabs twice
- **Файл**: `bot/handlers/search.py:578`
- **Проблема**: aiogram 3 dispatches each callback_query as its own concurrent task (no FSM lock, no per-user serialization is configured anywhere — Grep for asyncio.Lock shows only client-singleton and connect locks). handle_confirm_grab / handle_grab_best / handle_force_grab all follow the pattern: read session via db.get_session(), run grab_release()/_execute_grab() (many awaits to *arr/qBittorrent), then db.delete_session() only at the very end (search.py:751). If the user double-taps 'Подтвердить' (or taps Confirm then Force-Grab) the two callbacks both read the SAME non-deleted session before either deletes it, so add_service.grab_movie_release/grab_series_release runs twice → the same release is push_release'd/grabbed twice, or qBittorrent add_torrent_url is called twice with the same magnet/url, or two ActionLog GRAB rows are written. On a Pi over slow Wi-Fi the grab takes seconds, widening the window massively.
- **Риск**: Duplicate library adds, duplicate torrents in qBittorrent, duplicate grab actions, confusing 'message is not modified' / double success messages.
- **Решение**: Add a per-user in-progress guard before doing any grab work. Either (a) keep a module-level `_grab_in_progress: set[int]` guarded by an asyncio.Lock and reject the second callback with callback.answer('Уже обрабатывается...'), or (b) atomically claim the session: delete (or mark) the session at the START of the handler (e.g. read-and-delete in one DB step) so the second concurrent read finds nothing and shows 'Сессия истекла'. Apply the same guard to handle_grab_best, handle_force_grab and handle_confirm_music_add.
- **Верификация**: CONFIRMED — Reproduced the double-grab race in the CURRENT code. aiogram is set up with a bare Dispatcher() (bot/main.py:265) — no FSM storage, no concurrency flags — so each callback_query update from the same user is processed as an independent asyncio task with full interleaving at every await. The grab handlers read session then delete only at the very end: handle_confirm_grab (search.py:592 get_session) → grab_release → _execute_grab, which calls add_service.grab_movie_release/grab_series_release (search.py:682/730) and db.log_action (691/740), and only then db.delete_session(user_id) at search.py:75
- **Статус**: [x] Исправлено (раунд 4, TDD)

### RACE-02: Shared SQLite connection: explicit BEGIN transactions interleave across concurrent coroutines
- **Файл**: `bot/db.py:286`
- **Проблема**: Database holds ONE aiosqlite.Connection opened with isolation_level=None (autocommit), and two writers issue raw `await self.conn.execute("BEGIN")` ... commit/rollback: save_search (db.py:286-315) and cleanup_old_searches (db.py:498-517). aiosqlite serializes individual statements on its single worker thread, but a multi-statement BEGIN..commit block yields the connection at every `await`. The periodic cleanup task (main.py:305 -> _periodic_cleanup -> cleanup_old_searches every 6h) and any user's search (process_search -> save_search) run on the same shared connection. If their BEGIN/commit windows overlap, the second `BEGIN` raises sqlite3.OperationalError 'cannot start a transaction within a transaction', OR one coroutine's commit/rollback prematurely terminates the other's transaction — committing partial/rolling back already-staged rows. Two concurrent user searches hit the same race.
- **Риск**: Intermittent 'cannot start a transaction within a transaction' surfaced to users as 'Поиск временно недоступен', or silent partial commits / lost search_results rows under concurrency.
- **Решение**: Serialize write transactions. Add an asyncio.Lock (e.g. self._write_lock) on Database and wrap every explicit BEGIN..commit/rollback block (save_search, cleanup_old_searches) in `async with self._write_lock:`. Alternatively replace manual BEGIN with aiosqlite's `async with self.conn` context-manager and still guard with a single write lock, since one shared connection cannot have two logical transactions open at once.
- **Верификация**: CONFIRMED — Verified against current code. bot/db.py opens ONE shared connection with isolation_level=None (autocommit) at db.py:46. Two methods issue explicit multi-statement transactions on that shared connection with NO mutual exclusion: save_search (db.py:286 `await self.conn.execute("BEGIN")` ... db.py:311 commit / db.py:313 rollback) and cleanup_old_searches (db.py:498 BEGIN ... db.py:513 commit / db.py:515 rollback). A grep confirms the only asyncio.Lock in db.py is self._connect_lock (db.py:33), used solely inside connect(); there is no self._write_lock or any guard around the BEGIN..commit blocks
- **Статус**: [x] Исправлено (раунд 4, TDD)
